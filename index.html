redis设计与实现————单机数据库

发表于 2017-03-03 | 分类于 redis |

服务器中的数据库

redis服务器将所有的数据库都保存在服务器状态，redis.h/rediserver结构的db数组里面，db数组额每一项都是一个redis.h/redisDb结构，每一个redisDb结构代表一个数据库。在初
始化服务器的时候，程序会根据服务器状态的dbnum属性，来决定应该创建多少个数据库。

struct redisServer{
//服务器数据库的数量
    int dbNum;
//一个数组，保存着数据库中所有的数据库
 redisDb *db;
};

dbNum属性的值由于服务器配置的database选项决定，默认情况下，该选项的值是16，redis会默认创建16个数据库。

切换数据库

每个redis客户端都有自己的目标数据库，每当客户端执行数据库读写命令时，目标数据库都会成为这些命令的操作对象。客户端通过select <index> 来切换数据库。

数据库键空间

redis 是一个键值对数据库服务器，服务器中的每一个数据库都是有一个redis.h/redisDb结构表示，其中redis结构的dict字典保存了数据库中的所有键值对，我们将这个字典称为键空间。

  • 键空间的键也就是数据库的键，每个键都是一个字符串对象。
  • 键空间的值，也就是数据库的值，每个值可以是一个字符串对象，列表对象，哈希表对象，集合对象，和有序集合对象中的任意一种redis对象。
  • 因为数据库的键空间就是一个字典，所以所有针对数据库的操作，增删改查，都是通过对键空间字典的操作来实现的。比如，set ,rpush，hset,del,get lrange,特殊的flushdb,randomkey,dbsize,exists,rename,keys等。

主要介绍一下，读写键空间时候，redis做的一些维护操作：

  • 在读取一个键时，（读写操作都要对键进行读操作），服务器会根据键是否存在，来更新服务器的键空间命中次数和和键空间miss次数，这两个值可以在命令info stats的keyspace_hits属性和keyspace_misses 属性中查看。
  • 在读取一个键后，redis 会更新键的LRU(最后一次使用)时间，这个值可以用于查看一个键的闲置时间，可以用 object idletime <key> 命令查看键的闲置时间。
  • 如果服务器在读取一个键发现键已经过期，那么服务器会先删除这个键，然后才执行余下的操作。
  • 如果有客户端使用watch命令监视了某个键，那么服务器会在这个键修改后，把这个键标记为脏（dirty）,从而让事务注意到这个键已经被修改。
  • 服务器每次修改一个键后，都会对脏键计数器的值增1，这个计数器会触发服务器的持久化和复制操作。

设置键的生存时间和过期时间

redis 有四个命令可以设置键的生存时间（键可以存在多久）和过期时间（键什么时候删除）

  • EXPIRE <key> <ttl> 将键的生存时间设置为ttl秒
  • PEXPIRE <key> <ttl> 将键的生存时间设置为ttl毫秒
  • EXPIREAT <key> <timestamp> 将键的过期时间设置为timestamp秒时间戳
  • PEXPIREAT <key> <timestamp > 将键的过期时间设置为timestamp毫秒时间戳

虽然有四个命令，其中前三个都是使用PEXPIREAT来实现的。
通过expire命令和pexpire命令，客户端可以以秒和毫秒精度为数据库中的某个键设置生存时间（Time To Live TTL）， 在经过指定的生存时间后，服务器就会自动删除生存时间为0的键。

与expire命令和pexpire命令类似，客户端可以通过expireat 和pexpireat 命令以秒和毫秒精度给数据库里某个键设置过期时间。
过期时间是一个unix时间戳；

问题：

  • 数据库如何保存键的生存时间和过期时间。
  • 服务器如何自动删除那些带有生存时间和过期时间的键。

保存过期时间

redisDb结构的expires字典保存了数据库中所有键的过期时间，称为过期字典。

过期字典的键是一个指针，这个指针指向键空间的某个键对象。
过期字典的值是一个 long long 类型的整数，这个整数保存了键所指向的数据库的键的过期时间 一个毫秒精度的Unix时间戳。

有设置键的过期时间，就有相应的移除命令，PERSIST <key> 移除键的过期时间，就是将过期字典中该键移除。

如果忘记了一个键还有多少时间过期，redis也有相应的命令，TTL 和 PTTL 分别返回秒和毫秒单位的时间。
如果数据库中不存在这个键，将返回 -2 ，如果数据库中这个键的并没有设置过期时间，将返回-1 ，否则返回 （过期时间 - 当前时间）

过期键的判定

其实就是扫描，过期字典中的键，利用当前时间，与过期时间进行比较。

过期键的删除策略

  • 定时删除：在设置键的过期时间的同时，创建一个定时器，让定时器在键的过期时间来临时，立即执行对键的删除操作。
  • 惰性删除：放任键过期不管，每次读取键时，都检查取得的键是否过期，如果过期，就删除。
  • 定期删除：每隔一段时间，就对数据库进行一次检查，删除里面的过期键。至于删除多少过期键，以及要检查多少个数据库，择则有算法决定。

第一种和第三种为主动删除策略，第二种为被动删除策略。

优缺点：

定时删除：

  • 优点：对内存友好，可以保证冷数据及时的呗清除，释放过期键占用的内存。
  • 缺点：对cpu不友好，作为一个轻量，高速要求的内存数据库来说，如果过期键比较多的情况下，删除过期键这一行为，可能占用大量的cpu时间，将会大大影响正常的业务请求的响应时间和吞吐量。是redis所不能忍受的。而且，创建一个定时器需要用到redis服务器中的时间事件，时间事件的实现方式，是无序列表，查找的复杂度为O(N)，并不能高校的处理大量的过期键创建的时间事件。

惰性删除：

  • 优点:对cpu友好，延迟了那些无关键的删除操作，只针对当前处理的键，进行删除操作。
    • 缺点：对内存不友好,如果数据库中存在大量的过期键，而且以后可能不会在被访问到，这些键就会长久的占用内存，不会释放，（除非手动执行FLUSHDB）,可以被看做内存泄露。比如跟时间点有关的 日志（log） 在某个时间点后，访问大大减少，甚至不在访问，就会造成数据库大量数居的积压，内存得不到释放。

定期删除：

  • 定期删除是对上述两种方式的一种整合；定期删除，既不会让cpu长时间处于无关业务的事件上，同时又能定期的删除过期键，释放内存。
  • 但是如果定期删除执行的太频繁，执行的时间太长，就会退化成，定时删除。如果执行的次数太少，执行时间太短，又会和惰性删除策略类似，造成内存脏数据的积压。所以服务器必须根据业务数据，过期键的增长情况，合理的设置删除的次数和执行频率。

redis的过期键删除策略：

redis服务器使用惰性删除策略，和定期删除策略两种删除策略，服务器何以很好的在合理利用cpu,避免浪费内存空间之间取得平衡。实现：

  • 惰性删除策略：由db.c/expireIfNeeded函数实现，所有读写数据库键的命令都会在执行之前，调用expireIfNeeded函数对输入键进行检查。

  • 定期删除策略：
    过期键的定期删除策略有redis.c/activeExpireCycle 函数实现，每当redis周期性的操作 redis.c/serverCron 函数执行时，activeExpireCycle 函数就会被调用，他在规定的时间内，分多次遍历服务器中的各个数据库，从数据库中的expires字典中，随机检查一部分过期键的过期时间，并删除其中的过期键。
    大致就是，用一个全局变量当做游标，记录该次检查遍历的数据库的index， 从其中随机抽出固定部分键，检查是否过期，及过期键所占比例，规定一个比例阀值，直到随机抽查的键中，过期键占比低于阀值，便继续下一个数据库，同时更新游标值，以便下次改函数执行时，从游标记录的数据库开始执行，当执行到最后一个数据库时，所有的数据库遍历一遍，游标清0，下次函数执行，又从新开始遍历。

AOF RDB 和复制功能对过期键的处理

先抛出两个问题：

  • 如果数据库进行持久化，数据库中的过期键，要不要持久化。
  • 服务器重启时，载入持久化文件进入内存时，那些已经过期的键该如何处理，在主从复制模式下，又会怎么样。

生成RDB文件

在执行SAVE 命令，和 BGSAVE命令创建一个新的RDB文件时，程序会对数据库中所有的键进行检查，已经过期的键不会被保存到新创建的RDB文件中。因此数据库中的过期键，不会对生成新的RBD文件造成影响。

载入RBD文件

在启动服务器时，如果服务器开启了RDB功能，那么服务器将对RDB文件进行载入：

  • 如果服务器以主服务器模式运行，在载入文件时，程序会对文件中保存的键进行检查，未过期的键会被载入内存，过期键，会被忽略。所以RDB中过期的键对主服务器不会造成影响。
  • 如果服务器以从服务器的模式运行，在载入文件时，文件中所保存的键，无论是否过期，都会载入到数据库。因为数据库在进行主从同步数据时，从数据库的数据都会被清除，以主服务器数据为准。

AOF文件写入

  • 当服务器以AOF持久化模式运行时，如果数据库中的某个键已经过期，但他还没有被惰性删除，或者定期删除，那么AOF不会因为这个过期键而产生影响。
  • 当过期键被惰性删除，或者定期删除，程序会向AOF文件追一条DEl命令，来显示的标记该键已经被删除。
  • 比如客户端执行 get msg,而msg键此时已经过期，redis会先从数据库删除该键，向AOF 文件追加一条删除命令，向客户端返回空。

AOF 文件重写

和生成AOF文件类似，重写之前，会先检查数据库中的键，已过期的键不会重写到AOF 文件中去。

复制

当数据库运行在复制模式下，从服务器的过期键删除动作由主服务器控制。

  • 当主服务器，删除了一个过期键，回向所有他的小弟们，发送一个DEl命令，告知从服务器，删除了这个过期键。
  • 从服务器在执行客户端的读操作时，即使碰到过期键，也不会执行过期键的删除操作，而是继续像没过期的键一样进行处理。
  1. 通过主服务器来控制从服务器统一的删除过期键，可以保证主从服务器的数据一致性，也正是因为这个原因，当一个过期键存在主服务器时，这个过期键的复制品，也一样存在从服务器，从而保证主从服务器的数据的一致性。
  2. 如果这时有客户端向从服务器发送GET命令，那么从服务器同样会先执行键的检查，将发现message键已经过期，但是从服务器不会执行过期键的删除策略，而是继续将键的值返回给客户端。
  3. 假设有客户端向主服务器发送GET命令，主服务器先检查键是否过期，发现过期，就会执行过期删除策略，向客户端返回空，同时向从服务器发送DEL命令。
  4. 从服务器收到主服务器发来的Del命令，也会进行删除操作，主从服务器又会保持一致。

RDB持久化

redis 作为一个内存数据库，一旦redis进程意外退出，断电，关机，内存数据就会清空，所以必须有一个数据备份的机制，启动服务的时候能够把数据load进内存，运行时能够定期的把内存某一时期的快照进行持久化存储。redis通过 RDB 和AOF 两种方式实现。可以手动执行，也可以配置文件定期执行。

RDB 文件，保存了某个时间点，redis服务器中整个数据库的状态，数据。 redis通过SAVE 和 BGSAVE 命令来手动的触发RDB持久化。

问题：SAVE 和 BGSAVE的区别?

  • SAVE 命令会阻塞服务器进程，知道RDB文件创建完毕为止，在服务器进程阻塞期间，服务器不能处理任何命令请求；

  • BGSAVE 不会阻塞服务器进程，BGSAVE会派生出一个子进程。然后由子进程负责创建RDB文件，服务器进程继续处理，命令请求。
    redis服务器在启动时如果检测到RDB 文件，会自动的载入RDB文件。

  • 但是：如果服务器同时开启了AOF 和 RDB 功能,由于AOF的更新频率比RDB的更新频率高，服务器启动时，会优先检测AOF是否开启，如果开启，默认使用AOF 文件还原数据库状态。

BGSAVE 执行期间服务器的状态：

BGSAVE 执行期间父进程继续处理服务器的请求，子进程处理RDB持久化，如果这个时候，客户端在发送 SAVE ,BGSAVE ,BGREWRITEAOF 命令会怎么样。

  • SAVE ，直接拒绝，因为SAVE 命令由父进程执行，RDB 持久化 会调用同一个函数，rdbSave ，防止两个进程产生竟态条件。
  • BGREWRITEAOF 和 BGSAVE 不能同时执行。BGSAVE 正在执行，BGREWRITEAOF 会延迟到BGSAVE执行完毕才执行。BGREWRITEAOF执行期间，BGSAVE 会被直接拒绝。因为两个命令都会派生一个子进程进行持久化，并且两个子进程都会进行大量的磁盘写入操作，会对服务器的性能造成影响。

自动间隔保存

因为BGSAVE 可以在不阻塞服务器的情况下执行，所以redis允许用户可以设置服务器配置的save选项，让服务器每间隔一段时间自动执行一次BGSAVE；

每当数据库启动，就会把配置文件中的参数，load进服务器状态 redisServer 结构中的saveParams中；

saveparams 属性是一个数组，数组中的每一个元素都是一个 saveparam 结构，每一个saveparam 结构都把保存了一个save 选项设置的保存条件。

dirty计数器 和lastsave 属性

  • dirty 记录距上一次成功save 后，服务器执行的修改次数；
  • lastsave 记录上一次成功进行保存后的Unix时间戳；

服务器的常规性维护操作，时间事件，serverCron 其中一项就是检查save设置的保存条件是否满足，就是取服务器状态中的dirty 与saveparam 中的changes进行比较。还有当前时间减去服务器状态中的lastsave 时间和saveparam 中的seconds 比较。

AOF 持久化

AOF持久化的实现

与RDB持久化通过保存数据库中的键值对来记录数据库的状态不同，AOF 持久化是通过保存Redis服务器的写命令来记录数据库的状态的。

客户端 ——》 发送写命令 ——》 服务器 –> 保存被执行的写命令 ——》 AOF 文件

命令追加

当AOF 持久化功能处于打开状态时，服务器在执行完一个写命令后，会以协议格式，将执行的命令追加到服务器状态的aof_buf 缓冲区的末尾；

AOF 文件的写入和同步

redis服务器进程就是一个事件循环，这个循环中的文件事件负责接收客户端的请求，以及向客户端发送命令回复，而时间事件，负责执行像 serverCron 函数这样需要定时运行的函数；

服务器在执行文件事件时，可能执行一些写命令，使得一些内容被追加到服务器状态aof_buf 缓冲区，所以在服务器每次循环事件执行前，他都会调用 flushAppendOnlyFile函数，考虑是否将缓冲区的数据斜土和保存到AOF文件。

flushAppendOnlyFile 函数的行为有服务器配置的appendfsync 选项的值来决定；

  • always
  • everysec
  • no 一旦宕机，将丢失上次同步之后操作系统最后一次同步之后所有的数据

aof 文件的载入和还原

因为aof文件记录的是reids 写命令，而redis 的命令只能在客户端上下文中执行，而载入AOF 文件所使用的命令直接来源于AOF文件而不是网络连接，所以服务器创建了一个没有网络连接的伪客户端。来执行从AOF文件中分析出来的命令。

AOF 重写

问题：为什么要重写？重写时以什么为参照物？重写的文件有什么好处？

  • 因为Aof 持久化是通过保存被执行的命令，来记录数据库状态的，所以随着服务器运行时间的流逝，AOF 文件中的内容会越来越多，提及越来越大，不加以控制，会对服务器的运行造成影响，而且利用Aof 的还原数据非常耗时。
  • 重写时，并不以老的AOF 文件为基准，而是重新读取数据库状态，创建一个新的AOf 文件。来替换老的AOF文件。来减少那些冗余命令所占空间。
  • 假如一个集合，每次增加操作，往里面push 一个数据，最后这个集合一共有十万条数据，那就是十万条命令，要存到AOF文件里面去。但是重写后，单条命令 SADD 命令可以一次添加 64 个元素，原来的十万条命令，重写后，缩小了64倍。

AOF 后台重写

AOF从写程序 aof_rewrite函数会进行大量的写入操作，redis作为一个单线程程序，如果主线程来调用这个函数，势必会造成线程的长时间阻塞，这是redis所不允许的，和RDB 一样AOF重写，也会调用子线程来进行重写操作。但是有稍微区别，AOF 后台重写，并不是全程无阻塞；

当子线程开启后，所有客户端的写命令，都会追加到aof缓冲区和aof重写缓冲区，子线程重写完后，向父进程发送一个信号函数，父进程进入阻塞状态，并将aof重写缓冲区数据刷入新的AOF文件，原子的替换掉现有的AOF文件。

01-简单动态字符串 SDS simple-dynamic-string

redis 是由C语言实现的，但是并没有直接使用C语言定义的字符串（以空字符结尾的字符数组），而是自己构建了一种名为sds字符串的抽象类型。并将sds作为redis的默认字符串表示。
在redis里，C字符串只会用作字符串字面量，用在一些无需对字符串修改的地方，比如日志打印。在redis中，<K,V> 的K和V都是字符串，或者V是字符串列表的情况下，对象的底层都是存储的sds类型的字符串。

sds的定义

遵循这一惯例，主要是能够复用c字符串函数库里的一部分字符串函数。

sds与C字符串的区别

No C字符串 SDS
1 获取字符串长度的复杂度为O(N) 获取字符串长度的复杂度为O(1)
2 API是不安全的，可能造成缓存区溢出 API是安全的，不会造成缓存区溢出
3 修改字符串长度N次必然需要执行N次内存重分配 修改字符串长度N次最多需要执行N次内存重分配
4 只能保存文本数据 可以保存文本和二进制数据
5 可以使用所有<string.h>库中的函数 可以使用一部分<string.h> 库中的函数
  1. C字符串不记录自己的长度，每次需要遍历字节数组，sds通过属性len记录自己长度，实现O(1)复杂度获取字符换长度。所以对于那些频繁执行STRLEN的命令，并不会成为redis的瓶颈。
  2. C字符串，每次进行字符串的增长和缩短操作，都要对这个字符串数组进行内存的重分配。否则很容易造成缓冲区的溢出和内存泄漏。redis经常被用于速度要求严格，数据被频繁修改的场合，如果每次修改字符串都需要进行一次内存重分配，那么光是进行内存从分配就会在占用修改字符串这一操作大部分时间。为了避免C字符串的这种缺陷，sds通过未使用空间解除了字符串长度和底层数组长度之间的关系。通过未使用空间，sds实现了空间预分配，和惰性空间释放两种优化策略。空间预分配遵循一公式，当修改完后字符串的长度 len属性小于1MB,那么程序分配和len 相同长度的free预留空间，如果修改后长度大于等于1MB，将预先分配1MB的free预留空间。通过这一策略，redis可以减少连续增长字符串所需的内存分配次数。惰性空间释放刚好相反，当进行字符串缩短的操作时，并不立即进行缩短那部分的内存释放，而是增加free字段，作为未使用空间记录下来，减少了一次内存重分配，并将为将来可能的字符串增加提供了优化。以空间换速度。同时为了避免内存过度的消耗，sds提供了api,可以在有需要时，真正的释放sds的未使用空间。
  3. C字符串的字符串必须符合某种编码，并且字符串中除了末尾，不能含有空字符串，否则就会被程序误认为是字符串的结尾，所以只能保存文本数据，不能保存图片，音频，压缩文件这样的二进制数据。为了确保redis能够适应不同的场景，redis的api都是二进制安全的，所有SDS API都会处理二进制的方式处理SDS 存放在buf数组里的数据，程序写入什么样，读取就是什么样。SDS使用len属性而不是空字符来判断字符串的结束。

02-链表

链表在redis中的应用非常广泛，比如列表建的实现之一就是链表，当一个列表键包含了数量比较多的元素，或者列表中包含的元素都是比较长的字符串的时候，redis就会使用链表作为底层的数据实现。由于redis使用的C语言并没有内置链表这种数据结构，redis自己构建了链表的实现。

链表的定义

每个链表节点使用一个 adlist.h/listNode 结构实现

多个listNode可以通过prev和next组成双端链表，也就是双向链表；

虽然只需要listNode就可以组成链表，但是使用 adlist.h/list 来持有链表的的话，会更方便。

reids的链表实现的特性：

  • 双端 链表带有prev和next指针，能够O(1) 的复杂度，获取相邻节点
  • 无环，表头的prev和表尾的next指针都指向NULL,对链表的访问以null结束。
  • 带表头指针和表尾指针,能够很快的获取链表的头和尾节点信息，复杂度为O(1)
  • 带链表长度计数器，list结构的len属性持有链表的节点的长度，复杂度为O(1)
  • 多态 链表以void* 指针保存节点的值，并且可以通过 dup,free,match三个属性为节点值设定类型特定函数，所以链表能够保存各种不同类型的数据。

03 字典

  1. 字典，又称符号表（symbol table）。关联数组（associative array）或映射（map）,是一种用于保存键值对的抽象数据结构。
    字典中的每个key都是独一无二的，程序中能够根据键查找与之关联过的值，也可以通过键来国内新值，又或者根据键来更新值，或者删除值。
  2. 字典在redis中被广泛的应用，比如redis的数据库都是使用字典的作为底层实现的，对数据库的增删改查也是构建在对字典的操作之上的。
  3. 除了用来表示数据库之外，字典还是哈希键的底层实现之一，当一个哈希键包含的键值对比较多，又或者键值对中的元素都是比较长的字符串，redis就会使用字典作为哈希键的底层实现。

字典的实现

redis字典所使用的哈希表由dict.h、dictht结构定义：

table 属性是一个数组，数组中的每一个元素都是一个指向 dict.h/dictEntry 结构的指针，，诶个dictEntry结构保存着一个键值对。size 记录了哈希表的大小，也即是table数组的大小，而used则记录了哈希表目前已有节点的数量。sizemask属性的值总是等于size-1，这个值和哈希值一起决定了一个键应该被放到table数组的哪个索引上。

哈希表节点

哈希表节点由 dictEntry 结构表示，，每个dictEntry 结构都表示一个键值对：

key 属相保存了键值对中的key;而v属相则表示键值对中的值，用 union 联合体结构，表示值可以是一个指针，也可以是uint64_t 的整数，或者int64_t 的整数，三者互斥；
next 属性是指向另一个哈希表节点的指针，这个指针可以将多个哈希值相同的键值对连接在一起，以此来解决键冲突额问题。

mysql性能优化之explain

发表于 2017-02-04 | 分类于 mysql |

###使用explain语句去查看分析结果

如explain select * from test1 where id=1;

id    |    select_type    |    table    |    type | possible_keys    |    key |key_len | ref | rows | extra
  • type=const表示通过索引一次就找到了；
  • key=primary的话，表示使用了主键；
  • type=all,表示为全表扫描；
  • key=null表示没用到索引。
  • type=ref,因为这时认为是多个匹配行，在联合查询中，一般为REF。

###MYSQL中的组合索引

假设表有id,key1,key2,key3,把三者形成一个组合索引，则

如：

  • where key1=….
  • where key1=1 and key2=2
  • where key1=3 and key3=3 and key2=2

根据最左原则，这些都是可以使用索引的，如from test where key1=1 order by key3，用explain分析的话，只用到了normal_key索引，但只对where子句起作用，而后面的order by需要排序。

###使用慢查询分析

在my.ini中：

long_query_time=1

log-slow-queries=d:\mysql5\logs\mysqlslow.log

把超过1秒的记录在慢查询日志中

可以用mysqlsla来分析之。也可以在mysqlreport中，有如

DMS分别分析了select ,update,insert,delete,replace等所占的百份比

###MYISAM和INNODB的锁定

myisam中，注意是表锁来的，比如在多个UPDATE操作后，再SELECT时，会发现SELECT操作被锁定了，必须等所有UPDATE操作完毕后，再能SELECT

innodb的话则不同了，用的是行锁，不存在上面问题。

###MYSQL的事务配置项

innodb_flush_log_at_trx_commit=1

表示事务提交时立即把事务日志写入磁盘，同时数据和索引也更新。

innodb_flush_log_at_trx_commit=0

事务提交时，不立即把事务日志写入磁盘，每隔1秒写一次

innodb_flush_log_at_trx_commit=2

事务提交时，立即写入磁盘文件（这里只是写入到内核缓冲区，但不立即刷新到磁盘，而是每隔1秒刷新到盘，同时更新数据和索引

###explain用法

EXPLAIN tbl_name或：EXPLAIN [EXTENDED] SELECT select_options

前者可以得出一个表的字段结构等等，后者主要是给出相关的一些索引信息，而今天要讲述的重点是后者。

举例

mysql> explain select * from event;
+—-+————-+——-+——+—————+——+———+——+——+——-+
| id | select_type | table | type | possible_keys | key | key_len | ref | rows | Extra |
+—-+————-+——-+——+—————+——+———+——+——+——-+
| 1 | SIMPLE | event | ALL | NULL | NULL | NULL | NULL | 13 | |
+—-+————-+——-+——+—————+——+———+——+——+——-+
1 row in set (0.00 sec)
各个属性的含义

id

select查询的序列号

select_type

select查询的类型，主要是区别普通查询和联合查询、子查询之类的复杂查询。

table

输出的行所引用的表。

type

联合查询所使用的类型。

type显示的是访问类型，是较为重要的一个指标，结果值从好到坏依次是：

system > const > eq_ref > ref > fulltext > ref_or_null > index_merge > unique_subquery > index_subquery > range > index > ALL

一般来说，得保证查询至少达到range级别，最好能达到ref。

possible_keys

指出MySQL能使用哪个索引在该表中找到行。如果是空的，没有相关的索引。这时要提高性能，可通过检验WHERE子句，看是否引用某些字段，或者检查字段不是适合索引。

key

显示MySQL实际决定使用的键。如果没有索引被选择，键是NULL。

key_len

显示MySQL决定使用的键长度。如果键是NULL，长度就是NULL。文档提示特别注意这个值可以得出一个多重主键里mysql实际使用了哪一部分。

ref

显示哪个字段或常数与key一起被使用。

rows

这个数表示mysql要遍历多少数据才能找到，在innodb上是不准确的。

Extra

如果是Only index，这意味着信息只用索引树中的信息检索出的，这比扫描整个表要快。

如果是where used，就是使用上了where限制。

如果是impossible where 表示用不着where，一般就是没查出来啥。

如果此信息显示Using filesort或者Using temporary的话会很吃力，WHERE和ORDER BY的索引经常无法兼顾，如果按照WHERE来确定索引，那么在ORDER BY时，就必然会引起Using filesort，这就要看是先过滤再排序划算，还是先排序再过滤划算。

redis 背景

###首先要清楚三个问题

  • redis是谁搞出来的（装逼专用）
  • redis是个神马玩意？
  • 为什么会出来这么个玩意？
  • 这玩意为什么会迅速的火起来的？

2008年，意大利的一家创业公司 Merzia 推出了一款基于mysql的网站实时统计系统 LLOOGG,然而没过多久该公司的创始人 Salvatore Sanfilippo 便开始对mysql的性能感到失望，这家伙也是艺高人胆大，亲自为 LLOOGG量身定做了一个数据库，2009年开发完成，这个数据库就是redis，并大发善心，将redis开源，很快在程序员界圈粉无数，国内大量的互联网公司相继使用，例如新浪微博，知乎，街旁网等。国外的github，暴雪，等一大批粉丝。圈粉狂魔也不为过，小编我也没逃脱它的魔抓。

Redis 是一个高性能的key-value数据库。主要是存储数据，纳尼，数据库小编可是知道不少，比如mysql,oricle 这两个风靡全球的数据库。这就要说一下redis与mysql区别了。最本质的区别是两者数据存放的位置，和 存储的数据结构。mysql这类关系型数据库数据主要存储在本地磁盘，且存储的都是结构型数据，众所周知的mysql中的那一张一张的表，每个表每一行的数据结构都是固定的。redis主要把数据存储在内存中，也可持久化存储，或者log存储，且存储数据格式为非结构化数据。

redis 的出现主要解决的是，伴随着互联网的高速发展，大数据量，高并发，越来越多的非结构话数据的存储需求，传统的关系型数据库大量的Io读取，性能很快出现瓶颈，越来越无法满足web2.0 频繁的页面交互。一些场景无法满足。而redis 灵活的数据模型，高性能的读写能力，良好的扩展性 极大的满足了关系型数据库的一些不足。
